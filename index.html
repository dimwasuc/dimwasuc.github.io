2025.05 - 2025.06 | 10개 팀 중 1등

Python AWS Lambda SQS/SNS MySQL CloudWatch

기술적 도전

처음으로 AWS 서버리스 아키텍처를 활용한 프로젝트를 진행하면서 Lambda 함수 간 통신과 비동기 처리에 어려움을 겪었습니다.

특히 감정 분석 AI 모델 처리 시간이 길어 API Gateway의 30초 타임아웃 제한에 걸리는 문제와, 팀원들이 분산된 Lambda 함수들의 실행 상태와 에러를 실시간으로 파악하기 어려운 문제가 있었습니다.

해결방법

SQS를 도입하여 사용자 요청과 AI 처리를 분리하는 이벤트 기반 아키텍처로 전환했습니다. API Gateway는 즉시 응답을 반환하고, SQS 메시지를 통해 Lambda가 비동기적으로 AI 처리를 수행하도록 구현했습니다.

또한 SNS와 Slack Webhook을 연동하여 Lambda 실행 상태, 에러 로그, 처리 완료 알림을 팀 채널로 실시간 전송하는 모니터링 시스템을 구축했습니다.

CloudWatch 스케줄러를 활용해 매일 새벽 1시에 전체 사용자의 감정 벡터를 분석하여 커뮤니티 클러스터를 자동으로 재구성하는 배치 처리도 구현했습니다.

새로 배운 기술

AWS의 이벤트 기반 서비스들(SQS, SNS, EventBridge)을 활용한 느슨한 결합 아키텍처 설계 방법을 습득했습니다.

또한 서버리스 환경에서의 콜드 스타트 최소화, Lambda 함수 간 효율적인 데이터 전달 방식 등 클라우드 환경에 대해 더욱 깊게 이해할 수 있는 경험이었습니다.

실시간 영상 감상 및 채팅 커뮤니티 플랫폼

2025.06 - 진행중 | UMC 8기 협업 프로젝트

Node.js TypeScript WebSocket bcrypt GitHub Actions

기술적 도전

기존에는 구글, 카카오 등 소셜 로그인만 사용해왔는데, 이번 프로젝트에서 자체 로그인 시스템을 처음 구현하면서 보안과 인증 처리에 여러 이슈가 발생했습니다.

특히 비밀번호의 안전한 저장 방식과 사용자 인증 토큰 생성에서 보안을 고려한 구현이 까다로웠습니다.

해결방법

비밀번호 보안을 위해 bcrypt 라이브러리를 도입하여 salt 값과 함께 해시화하여 데이터베이스에 저장했고, 사용자 인증 토큰은 Node.js의 crypto 모듈을 활용해 crypto.randomBytes(32).toString('hex') 방식으로 예측 불가능한 64자리 랜덤 문자열을 생성하도록 구현했습니다.

생성된 토큰은 데이터베이스에 저장하여 사용자 인증 시 검증하는 방식을 사용했습니다.

새로 배운 기술

GitHub Actions를 활용한 CI/CD 파이프라인을 구축하면서 main 브랜치에 코드 푸시 시 자동으로 테스트, 빌드, EC2 서버 배포까지 이어지는 자동화 프로세스를 경험했습니다.

이를 통해 수동 배포 과정에서 발생할 수 있는 실수를 방지하고 배포 시간을 단축할 수 있었습니다.

Azure Co-op Translator 오픈소스 기여

Open Source Contribution | PR #166 (Issue #46)

Python Black Formatter Open Source

기술적 도전

Azure Co-op Translator의 세 개의 핵심 번역 모듈(MarkdownTranslator, ProjectTranslator, ImageTranslator)의 에러 메시지가 사용자에게 충분한 정보를 제공하지 못하는 문제를 발견했습니다.

특히 번역 실패 시 파일명이 없어 어떤 파일에서 문제가 발생했는지 알 수 없었고, 해결 방법에 대한 안내가 부족했습니다.

해결방법

Python 코드를 수정하여 모든 에러 메시지를 'Action failed for filename: Specific cause. Resolution hint.' 패턴으로 통일했습니다.

MarkdownTranslator에서는 타임아웃 에러와 LLM 프로바이더 설정 메시지에 파일명을 추가했고, ProjectTranslator에서는 Computer Vision API 키 설정 관련 환경 변수(AZURE_COMPUTER_VISION_KEY) 가이드를 명시했습니다.

ImageTranslator에서는 텍스트 감지 실패 시 폰트 설치 안내 등 구체적인 해결책을 제시했습니다. Black 포매터를 적용하여 코드 스타일을 프로젝트 컨벤션에 맞췄습니다.

새로 배운 기술

오픈소스 프로젝트의 코드 리뷰 프로세스를 경험하며 협업의 중요성을 배웠습니다.

리뷰어의 피드백에 따라 테스트 코드 구조를 여러 차례 수정하고, CRLF/LF 라인 엔딩 이슈를 해결하는 등 프로젝트의 일관성을 유지하는 방법을 익혔습니다.

특히 기존 코드 로직을 변경하지 않으면서도 사용자 경험을 크게 개선할 수 있다는 점과, 명확한 에러 메시지가 개발자 경험에 미치는 영향을 체감했습니다. 또한 Black 포매터 같은 도구를 활용한 일관된 코드 스타일 유지의 중요성을 배웠습니다.
